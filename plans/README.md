## Dependency notes

- Before executing 002-008, land the current `src/components/ui/Button.tsx`
  and `src/components/ui/Dropdown.tsx` follow-up. Executors must reuse these
  primitives and must STOP rather than recreate local button, popover, or menu
  implementations if they are absent from their checkout.
- 001 restores the existing green-gate contract before broader UI work.
- 002 establishes when controls may appear and how real native work is
  supplied; it standardizes single actions on `Button`/`ActionButton`. Plans
  003-007 must use that contract and the shared UI primitives rather than
  inventing adapters or local `Pressable` controls.
- 003 is now a consumer migration to `Dropdown`; it does not own another menu
  abstraction. Later plans may style or position consumers through the public
  primitive props, but must not fork its dismissal/focus lifecycle.
- 004 owns the shared modal surface. Table and Mermaid plans must not fork
  separate fullscreen implementations.
- 008 is last because its reviewed screenshots must represent the completed
  behavior, not bless known broken states.

## Findings considered and rejected

- Bundle Expo Clipboard/FileSystem/Sharing, Shiki, Mermaid, or a WebView in
  the core entry: rejected because optional-provider isolation is an explicit
  package and Metro gate. Plan 002 keeps host capabilities explicit.
- Copy browser Tailwind dimensions exactly: rejected because React Native
  needs 44-point touch targets and has no primary hover state. Match hierarchy,
  spacing, colors, and behavior while preserving native accessibility.
- Treat the pinned 1,511-case ledger as proof of visual parity: rejected;
  marker presence and semantic unit tests do not verify rendered device UI.
- Add a new design-system dependency: rejected. Existing theme tokens,
  React Native primitives, and `react-native-svg` are sufficient.

## Not audited

This focused assessment did not re-audit image downloading, math rendering,
general Markdown parsing, dependency currency, or the complete security
surface. Mermaid sanitization and offline WebView boundaries were reviewed
only far enough to ensure the proposed UI work does not weaken them.
